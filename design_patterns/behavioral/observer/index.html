Observer Pattern

Define a one-to-many dependency so that when one object changes state, all its dependents are notified

What is the Observer Pattern?

The Observer pattern is a behavioral design pattern that defines a one-to-many dependency between objects. When the state of one object (the Subject) changes, all its dependents (Observers) are notified and updated automatically.

This pattern is widely used in implementing distributed event handling systems and is fundamental to many reactive programming frameworks. It provides a way to react to state changes in objects without coupling the Subject to its Observers.

Key Participants

  • Subject: Knows its observers and provides an interface for attaching and detaching observers
  • ConcreteSubject: Stores state of interest to observers and sends notifications when state changes
  • Observer: Defines an updating interface for objects that should be notified of changes
  • ConcreteObserver: Implements the Observer updating interface to keep its state consistent with Subject's

Implementation

Here's a JavaScript implementation of the Observer pattern:

// Observer interface
class Observer {
    update(subject) {
        throw new Error('Observer.update() method must be implemented');
    }
}

// Subject (Observable) interface
class Subject {
    constructor() {
        this.observers = [];
    }
    
    attach(observer) {
        if (observer instanceof Observer) {
            this.observers.push(observer);
        } else {
            throw new Error('Observer must implement Observer interface');
        }
    }
    
    detach(observer) {
        const index = this.observers.indexOf(observer);
        if (index !== -1) {
            this.observers.splice(index, 1);
        }
    }
    
    notify() {
        for (const observer of this.observers) {
            observer.update(this);
        }
    }
}

// Concrete Subject
class StockMarket extends Subject {
    constructor() {
        super();
        this.symbol = 'ACME';
        this.price = 100.00;
        this.previousPrice = 100.00;
    }
    
    getSymbol() {
        return this.symbol;
    }
    
    getPrice() {
        return this.price;
    }
    
    getPriceChange() {
        return this.price - this.previousPrice;
    }
    
    updatePrice(newPrice) {
        this.previousPrice = this.price;
        this.price = newPrice;
        this.notify();
    }
}

// Concrete Observer
class Investor extends Observer {
    constructor(name) {
        super();
        this.name = name;
        this.logs = [];
    }
    
    update(stockMarket) {
        const price = stockMarket.getPrice();
        const symbol = stockMarket.getSymbol();
        const priceChange = stockMarket.getPriceChange();
        
        const direction = priceChange >= 0 ? 'up' : 'down';
        const message = `${symbol} price changed to $${price.toFixed(2)} (${direction} $${Math.abs(priceChange).toFixed(2)})`;
        
        this.logs.push({
            timestamp: new Date(),
            message: message
        });
    }
    
    getName() {
        return this.name;
    }
    
    getLogs() {
        return this.logs;
    }
}

When to Use the Observer Pattern

  • When changes to one object require changing others, but you don't know how many objects need to change
  • When an object should be able to notify other objects without making assumptions about those objects
  • When an abstraction has two aspects, one dependent on the other
  • When you need to maintain consistency between related objects without making them tightly coupled

Real-World Examples

  • Event handling systems: UI components notify listeners when user interactions occur
  • Newsletter subscriptions: Subscribers receive updates when new content is published
  • Social media notifications: Users receive updates when someone they follow posts content
  • MVC Architecture: Model notifies Views when data changes
  • Pub/Sub systems: Message brokers route events to appropriate subscribers

Benefits

  • Supports loose coupling between the subject and its observers
  • Allows sending data to other objects without knowing their specific types
  • Supports the Open/Closed Principle by allowing new observers to be added without modifying the subject
  • Subjects and observers can vary and evolve independently

Drawbacks

  • Observers are notified in random order
  • Potential memory leaks if observers aren't properly detached
  • Can lead to complex update dependencies that are hard to track
  • May cause cascading updates if not implemented carefully
