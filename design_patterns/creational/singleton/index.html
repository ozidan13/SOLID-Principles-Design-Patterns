Singleton Pattern

Ensure a class has only one instance and provide a global point of access to it

What is the Singleton Pattern?

The Singleton Pattern is a creational design pattern that ensures a class has only one instance while providing a global point of access to this instance. It's one of the simplest design patterns and involves only a single class which is responsible for creating its own instance.

When to use it:

  • When exactly one instance of a class is needed throughout the application
  • When the instance should be accessible at a well-known access point
  • When lazy initialization or efficient resource usage is required

Common examples:

  • Configuration managers
  • Database connection pools
  • Logger instances
  • File managers
  • Thread pools

Implementation

Here's how to implement the Singleton pattern in JavaScript:

JavaScript ES6 Implementation
class Singleton {
    constructor() {
        // Check if an instance already exists
        if (Singleton.instance) {
            return Singleton.instance;
        }

        // Initialize properties
        this.id = Math.random().toString(36).substring(2, 10);
        this.timestamp = new Date().toISOString();
        this.data = { counter: 0 };

        // Store the instance
        Singleton.instance = this;
    }

    // Static method to get the instance
    static getInstance() {
        // Create an instance if it doesn't exist yet
        if (!Singleton.instance) {
            Singleton.instance = new Singleton();
        }
        
        return Singleton.instance;
    }

    // Instance methods
    incrementCounter() {
        this.data.counter += 1;
        return this.data.counter;
    }

    getInfo() {
        return {
            id: this.id,
            timestamp: this.timestamp,
            counter: this.data.counter
        };
    }
}

Alternative Implementation (Module Pattern)

Using Closure (Module Pattern)
const DatabaseConnection = (function() {
    // Private instance variable
    let instance;
    
    // Private constructor function
    function createInstance() {
        const id = Math.random().toString(36).substring(2, 10);
        const timestamp = new Date().toISOString();
        
        // The actual instance object
        const connectionObject = {
            id: id,
            timestamp: timestamp,
            connectionStatus: 'Closed',
            
            connect: function() {
                this.connectionStatus = 'Open';
                return `Connected to database with ID: ${this.id}`;
            },
            
            disconnect: function() {
                this.connectionStatus = 'Closed';
                return 'Database connection closed';
            },
            
            getStatus: function() {
                return `Connection ${this.connectionStatus} (ID: ${this.id})`;
            }
        };
        
        return connectionObject;
    }
    
    // The public API
    return {
        // This is the only way to get the instance
        getInstance: function() {
            if (!instance) {
                instance = createInstance();
            }
            return instance;
        }
    };
})();

Key Points

  • Single Instance: A singleton class can have only one instance throughout the application lifecycle.
  • Global Access Point: Provides a static method or property to access the singleton instance.
  • Lazy Initialization: The instance is typically created only when it's first requested.
  • No Parameters: The getInstance method typically requires no parameters, since it always returns the same instance.

Advantages

  • Ensures a class has just a single instance
  • Provides a global access point to that instance
  • Protects the instance from being overwritten
  • Controls resource sharing efficiently
  • Reduces the need for global variables

Disadvantages

  • Makes unit testing more difficult
  • Violates the Single Responsibility Principle (creation and business logic)
  • Can introduce global state issues if overused
  • Requires special treatment in a multithreaded environment
  • Can hide dependencies rather than making them explicit
